唉，这段代码干了什么，怎么想不起来了，早知道我就埋个点，写一些性能日志！

这不是调用了XX团队接口嘛，不知道它返回的是什么状态码，响应耗时是多少啊！

这日志打印得毫无章法，有价值的日志信息都被淹没了，真特么“头疼”！

    笔者常常听到这样的感叹，“唉，我们有日志系统，而且花了巨大的成本搭建，怎么就不能快速有效的解决线上问题呢？”，其实这些“痛”归因是应用程序日志工程化问题。

日志工程实践

日志分类

    下面以开发Web系统的视角出发，我们来看看日志的总体分类。

系统日志

    比如Linux操作系统运行日志，包含内核和许多程序会产生各种错误信息、警告信息和其他的提示信息（本文不重点阐述）。

编程语言日志

    Java、PHP、Golang、Python等语言级别日志，比如PHP的错误日志、慢请求日志、php-fpm错误日志等，需重点关注。

应用程序日志

    业务系统运行的各项日志，它大致分为两类：框架内部日志与业务日志。

其他日志

    MySQL慢请求日志，Redis服务进程日志及其他中间件日志，这些日志运维开发人员关注较多。

    本文重点简述语言级别日志与应用程序日志，它们和应用程序工程化体系密切相关，且工程开发人员应重点关注与设计的。

    古人云：“无规矩不成方圆。”，笔者深有体会，在实践架构工作中，“规范”二字尤其重要，在日志使用规范上达成统一的共识，从业务角度出发，规范所有业务系统的日志输出、收集、可视化、处理、分析和报警，从而达成以下架设目标：

· 分钟级定位线上故障；

· 服务可用性可量化；

· 服务质量报表（日、周）；

· 特定用户请求可追溯与跟踪；

· 业务接口级预警与告警

· …

日志规范

适用范围

    日志规范适用于公司内所有技术栈、开发框架，它约定了读写日志的标准，包含日志书写格式，字段名称等。

json格式

    所有类型日志输出json格式，包括但不局限于PHP应用访问日志、PHP 应用错误日志、Nginx访问日志、PHP慢日志、PHP错误日志、Java日志、PHP Eureka Agent日志，Golang应用访问日志。应用程序直接输出json格式的日志，可以节约大量的CPU资源，因为json格式的日志可以不需要通过filter程序直接写入kafka。

日志级别

日志类型

请求链路日志

    请求链路日志是指发生服务调用请求会落盘一条请求日志，它记录了调用的一些基本信息，如请求发出时间，响应耗时等，日志模板如下：

响应链路日志

    响应链路日志是指当前请求处理完成并响应服务调用方后落盘的日志，它记录了当前请求处理的一些基本信息，如请求到达时间，处理耗时，响应状态码等，日志模板如下：

其他日志

    其他日志是指除链路跟踪日志以外，业务需要手工记录的日志，也可追溯。

日志字段
time

    必选，请求到达时间或服务调用时间，格式如：2018-03-15T13:41:08.123456Z

level

    可选，日志级别，详见（日志级别）

cat

    必选，日志类型，链路日志值为t_req（链路请求）或t_rsp（链路响应），其他日志值为log

app_name

    必选，当前应用名称，通常为注册到微服务平台的名称+版本号

trace_id

    必选，请求标识即trace_id，它用来标识用户请求，而这个请求可能会经过n次服务器内部接口调用、RPC、转发等，通过trace_id能够确定请求的整个链路，以便于快速定位和复现问题，分析用户行为等。

upspan

    必选，请求的上游服务名

span_id

    必选，是一个多维序列。第一次链路的时候初始值是0，每进行一次深入调用的时候就变成0.1，然后再升就是0.1.1，它每进行一次同深度的调用，就是说A 调完B 以后又调了D 就会变成0.2。

rt

    必选，当前请求响应耗时或被调用方响应耗时，单位ms

e_uri

    必选，当前请求来源的入口uri

uri

    必选，当前请求的uri或被调用方的uri

code

    必选，当前请求的响应状态码或被调用方请求的响应状态码，即返回数据中的code字段值

protocol

    必选，协议标识，如http/thrift等

protocol_code

    必选，当前请求响应协议状态码或被调用方响应协议状态码

verb

    必选，当前请求的HTTP方法名或被调用方请求HTTP方法名

query

    可选，当前请求查询参数或被调用方查询参数，为空时值为-

ext

    可选，当前请求自定义和当前请求相关的其他日志信息或被调用方请求相关的其他日志信息，为空时值为{}

ext.mem

        可选，当前请求处理的内存消耗，为空时值为-

ext.user_agent

        可选，当前请求的http user-agent报头，为空时值为-

ext.remote_addr

        可选，当前远程客户端的来源IP，为空时值为-

ext.server_addr

        可选，当前处理当前请求的服务器IP，为空时值为-

profile

        可选，当前请求自定义的性能分析日志信息，为空时值为{}

msg

        可选，其他日志信息

调用链路图例

调用链路约束如下：

· trace_id随着每一次调用透传到下游服务；

· span_id第一次链路的时候初始值是0，每进行一次深入调用的时候就变成0.1，然后再升就是0.1.1，它每进行一次同深度的调用，如A调用B，然后B调完成C，又调了D就会变成0.1.2；

· 请求链路和响应链路日志可配对，且span_id，upspan两个字段值相等；

· 请求链路日志在请求发出前立即写入内存堆栈，待响应返回后填充其他字段后落盘（考虑异步/协程）；

报头字段

    要实现完整链路跟踪要求我们在发送服务请求时设置四个报头：

x-g7-trace-id

    即请求标识即trace_id，是调用方传递到被调用方；

x-g7-upspan

    即请求的上游服务名upspan，是调用方传递到被调用方；

x-g7-span-id

    即链路的spanid，是调用方传递到被调用方；

x-g7-e-uri

    即当前请求来源e_uri，是调用方传递到被调用方；

性能分析（埋点）

    原则上任何有网络io读写请求的位置均需要进行性能埋点，比如：

· 数据库组件

· 缓存组件

· 文件IO组件

请求链

请求链路

    在分布式系统中请求链（又称调用链）相当重要，它是跟踪用户请求的基本手段，同时也能绘制庞大的分布式系统的服务依赖。基于请求链可以迅速定位分布式系统中相应节点的故障，及时调整与恢复服务。

    一个前端（web程序/移动应用）请求到达入口服务（网关）时，首先就会产生一个trace_id，入口服务（网关）将trace_id以报头的形势向下游服务传递；

    下游服务在解析请求时，首先检查报头中是否已经有trace_id，如果有，则直接使用报头中的trace_id，否则按一定的规则产生一个trace_id，并继续向下传递，依次类推。

    最终响应前端请求时，将trace_id作为响应报头中的一部分一并输出。

    X-G7-Trace-Id报头即为请求响应的日志ID，通过此日志ID就可以在日志可视化平台，通过查询条件trace_id:7f015a2567f5004a4d3f，查询到所有和当前请求相关依赖服务的对应日志。

    对于前台web程序（pc/app）尤其重要，因为通常出现问题首先是最前端的用户、产品、测试同学发现问题，这个时候如果提供请求的日志ID，将能够有利于快速定位问题。

其他规范

什么时候将应用访问日志落盘？

    推荐的做法是在每个请求响应后，将日志刷到磁盘，各大语言可以在框架层面封装，应用开发可以透明使用，无须关心。

日志可以包含换行符吗？

    理论上日志文件内每行代表一条日志，多条日志也是通过换行符分隔，如果日志中有换行符，推荐通过json encode后再输出日志。

框架封装日志格式后，如何灵活扩展字段？

    推荐各大语言框架push log api，提供将请求上下文相关的信息一并写入应用访问日志。

基于ELK日志平台实践

    上面阐述了应用程序日志规范，下面我们来聊聊日志收集、检索、可视化等实践。

总体架构

    日志平台系统，负责收集所有系统日志、业务日志，转化为流式数据，通过logstash/flume/Filebeat等输入kafka集群，然后通过logstash、Storm、Spark等系统从kafka集群读取并实时分析处理日志，写入ElasticSearch提供数据查询，并将日志持久化存储到HDFS供离线数据分析处理，并且可以发起异常报警或提供性能指标。

日志数据流

日志收集

    部署在Docker的应用，只需要将日志输出系统的标准输出，由于容器的标准输出会与物理（虚拟）机的一个文件映射，故一台物理（虚拟）机部署一个logstash实例即可。

日志储存

    所有的日志收集程序将日志上报到kafka集群，kafka集群缓存日志，可以设置一定的缓存时间，通过日志处理程序从kafka消费日志，将日志索引后储存到ElasticSearch或者HDFS。

日志可视化

    日志储存ElasticSearch后，可直接通过kibana可视化

    日志检索通过Lucene Query String Syntax，语法规则可参考：

https://segmentfault.com/a/1190000002972420

实时日志监控面板

    通过大屏投放的形式，研发团队随时实时关注系统运行状态。从实际操作来看，研发人员往往在监控报警系统告警之前就发现系统故障，进而迅速排查解问题根源并解决。

工具目标：

  1. 实时关注反馈系统运行状态；

  2. 快速分析故障影响的核心系统；

实时TopN接口响应超时3日对比看板

    接口维度3日横向对比响应超过一定时间（秒）数量，快速分析接口是否异常，让“小白”也能看懂接口服务状态，如有异常，点击柱形图可跳转到Kibana，分析日志详情，进一步排查问题根源并解决。

工具目标：

  1. 直接暴露异常接口；

  2. 一键分析故障根源是由于哪些个依赖资源（系统）；

实时统计最近15分钟各服务器接口超时量

工具目标：
