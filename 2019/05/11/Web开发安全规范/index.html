Web开发安全规范

Web开发安全规范

一、概述

1.1 、目的

    保障 Web 应用平台的安全性、可用性、完整性和数据保密性。从安全角度规范 Web 应
用系统开发人员，提高开发人员的安全意识，在开发过程中进行安全编程。

1.2 、适用范围

    公司全体开发人员

二、具体实施

2.1 、web 开发安全规范条例

1）认证相关的密码（例如 SQL 连接密码、加密用的 Key 等）都要满足《密码安全规范》要
求。

2）对用户的操作要进行权限判定，避免被越权。

3）认证模块必须采用防暴力破解机制，例如登录、注册、敏感操作等功能模块，都要使用
验证码或其他二次验证方式。

4）所有的输入和输出数据都要进行安全编码转义或过滤。

5）不能依赖于客户端校验，必须使用服务端代码对输入数据进行最终校验。

6）敏感数据（例如登录注册时帐号密码）传输不能使用 GET（要使用 POST），并且进行加
密。

7）对数据进行合法验证时，要使用枚举或正则表达式等进行白名单校验（因为使用黑名单
校验容易被绕过）。

8）上传文件时需要在服务端对文件类型进行白名单校验，并且对文件进行重命名，上传路
径也不能由用户控制。

9）禁止将敏感文件（如代码备份文件、日志文件、配置文件、数据库文件、SVN 版本控制
文件等）存放在 Web 内容目录下，导致可被直接下载。

10）应用程序出现异常时，禁止向客户端暴露不必要的信息，只能向客户端返回一般性的错
误提示消息。

11）禁止将带有 phpinfo()等服务器信息的页面直接放在生产环境中。

2.2 、Web 漏洞类型与防范方法

SQL 注入漏洞

测试方法：

    在 URL 参数中提交分隔符和拼接 SQL 语句，通过回显或盲注查询出数据库中的其他信息。

防护方法：

  1. 针对数字型的注入，可使用 is_numeric()、ctype_digit()、intval() 或正则表达式进
    行过滤；
  2. 针对字符型的注入，可使用 addcslashes()、mysql_real_escape_string()进行过滤。
    XSS 漏洞
  3. 使用PDO参数绑定

XSS 漏洞

测试方法：

    在 html 输出点或留言版等存储型位置提交带有其他 script 脚本的语句，从而执
行窃取 Cookie 等恶意操作。

防护方法：

    在前端和后端都使用 htmlspecialchars()对”、’、<、>、等字符进行转义和过滤。

CSRF 漏洞

测试方法：

    先登录一个帐号，然后直接访问一个构造好的带操作请求的连接，可导致执行了
敏感的操作。

防护方法：

    敏感操作使用二次认证，或是限制 Referer，或是采取 token 验证。

上传漏洞

测试方法：

    直接上传 webshell；上传后缀黑名单以外同样可以解析的 webshell；上传 webshell
时抓包修改 content-type 从而绕过文件类型判断；利用%00 截断上传 webshell；将 webshell
和图片合并，欺骗文件类型判断从而上传。mine判断

防护方法：

    白名单机制检验文件后缀名；

    图片文件上传后使用 GD 库进行二次渲染；

    上传后将文件名重新命名；

    上传目录设置不解析网页文件。

文件包含漏洞

测试方法：

​ 修改 URL 中的文件名参数，或是通过../跳到其他目录去引用其他的文件【谨慎include，不要include用户输入的地址】，从而执行其他文件的命令或是输出私密文件的内容。

    

防护方法：

    采用白名单的方式限定可被包含的文件；限定通过../跳转到其他目录。

目录遍历漏洞

测试方法：

    直接访问 web 下的目录，可看到目录里面的文件列表。

防护方法：

    配置 web 服务器使其不直接列目录，并且一些敏感目录设置外部不可访问。

代码/ 命令执行漏洞

测试方法：

    在带有命令执行的参数中拼接其他命令语句或是其他代码，从而执行自定义的代码或命令。

防护方法：

    使用 escapeshellarg()函数过滤；

    php.ini 中，设置 safe_mode=On，safe_mode_exec_dir= 路径名来限制可执行命令的目录。

访问控制缺失

测试方法：

    不通过登录，直接访问管理页面，若是没做限制，可直接进行高权限的操作。
防护方法：

    控制好权限分配，高权限的操作都得通过身份验证。

URL 重定向漏洞

测试方法：

    修改 URL 中跳转到某个目标的参数，可跳到任意指定的目标 URL。

防护方法：

    进行 URL 跳转前先判断所要跳转的目标合不合法，采用白名单的方式设置可跳转的 URL。

HTTP 参数污染漏洞

测试方法：

    提交 http 请求时，设置同名的参数不同的值，可绕过程序逻辑的判断，比如绕过 WAF 拦截。

防护方法：

    设计程序判断逻辑时，考虑 HPP 场景，提取出正确的参数值。

配置不当漏洞

测试方法：

    针对特定的配置不当，去进行相应的利用操作。
防护方法：

    按照安全规范的标准去进行配置。

任意文件操作

测试方法：

    通过 URL 操作一个文件时，修改文件名参数，从而去操作权限以外的文件。

防护方法：

    进行文件操作前先判断权限，下载文件操作后后端进行输出，做好权限控制。

弱口令

测试方法：

    加载字典中常见的强度比较低的密码去尝试登录。

防护方法：

    使用强度比较大的密码，最好是长度是 8 位以上，并且是字母+数字+字符。

点击劫持

测试方法：

    在页面的输出点插入 HTML 代码，设置成透明 iframe，从而误导用户去点击，导
致执行一些危险的操作。

防护方法：

    通过设置 HTTP 头的 X-FRAME-OPTIONS 去限定允许 frame 加载的页面地址。

敏感信息/ 文件泄漏

测试方法：

    通过使用 DirBuster 等工具去猜解 web 目录，或是通过网站爬虫去抓取和分析出
一些敏感的信息和目录。

防护方法：

    隐藏好私密信息，可不对外的文件或信息要设置禁止外部访问。同时控制好访问的权限。

2.3 、业务逻辑漏洞类型与防范方法

1 、 注册

无限制批量注册 帐号

测试方法：

    注册一个帐号同时进行抓包，编写多线程脚本重放这个包（每个包都使用不同的
注册用户名）。

防护方法：

    对一个 IP 一定时间内允许的注册数做限制，并且增加验证码。

撞库扫号

测试方法：

    利用注册前判断用户是否已存在的接口，去验证其它库的用户在该平台是否也存在。

防护方法：

    注册前判断用户是否已存在的请求做次数限制，超过阈值，不管用户是否已存在，都返回存在。

验证码识别破解

测试方法：

    使用验证码识别工具识别验证码并自动输入。

防护方法：

    使用不易识别的验证码，或是使用拖拉型的验证码。

验证码复用

测试方法：

    注册时抓包，使用同一个验证码，把数据包重放，可以注册多个帐号。

防护方法：

    程序逻辑要求同一个验证码短时间只能用一次或几次，不管验证是否通过都刷新。

验证码绕过

测试方法：

    修改 URL 或 cookie 中的某个参数，可无需验证码直接批量注册多个帐号。

防护方法：

    程序逻辑要求任意的注册请求数超过阈值之后都要启用验证码。

验证码明文输出在客户端

测试方法：

    查看 HTML 代码或 Cookie 可看到明文的验证码

防护方法：

    验证码应保存在 session 中进行校验，而不能输出在 HTML 或 Cookie 中。

2 、 登录

无限制密码尝试次数

测试方法：

    登录一个帐号同时进行抓包，编写多线程脚本重放这个包（每个包都使用不同的帐号或密码）。

防护方法：

    对一个 IP 一定时间内允许的登录尝试次数做限制，并且增加验证码。

    一个账号密码错误达到一定次数后，进行一定时间的账号锁定，不允许登录

撞库

测试方法：

    使用其他平台被拖的库的帐号密码来尝试登录。

防护方法：

    对一个 IP 一定时间内允许的登录尝试次数做限制，并且增加验证码。

验证码识别破解

测试方法：

    使用验证码识别工具识别验证码并自动输入。

防护方法：

    使用不易识别的验证码，或是使用拖拉型的验证码。

验证码复用

测试方法：

    登录时时抓包，使用同一个验证码，把数据包重放，可以注册多个帐号。

防护方法：

    程序逻辑要求同一个验证码短时间只能用一次或几次。

验证码绕过

测试方法：

    修改 URL 或 cookie 中的某个参数，可无需验证码直接批量登录多个帐号或是对帐号进行密码爆破。

防护方法：

    程序逻辑要求任意的登录请求数超过阈值之后都要启用验证码。

验证码明文输出在客户端

测试方法：

    查看 HTML 代码或 Cookie 可看到明文的验证码。

防护方法：

    验证码应保存在 session 中进行校验，而不能输出在 HTML 或 Cookie 中。

3 、 密保设置

CSRF （跨站请求伪造）

测试方法：

    先登录一个帐号，然后直接访问一个构造好的带操作请求的连接，可导致执行了设置密保的操作。

防护方法：

    密保设置操作使用二次认证，或是限制 Referer，或是采取 token 验证。

验证步骤绕过

测试方法：

    进行验证时，随便输入，并且拦截服务器返回的包进行修改，欺骗浏览器（前端）使其以为通过验证，从而绕过验证的步骤。

防护方法：

    判断是否通过验证，应该在后端进行判断，而不是在前端进行判断。

设置他人帐号的密保

测试方法：

    设置密保时，抓包将自己帐号 ID 改成其他用户的，去设置他人的密保。
防护方法：

    控制平行权限，禁止越权设置他人的密保。

邮箱/ 手机短信炸弹

测试方法：

    调用发送邮件/短信的接口时进行抓包，写脚本去不断重放这个数据包，可无限制的发送大量邮件或短信。

防护方法：

    对同一个邮箱/手机号一定时间内能收到的邮件个数/短信条数进行限制。

    验证接口请求ip，做限流

    统计单个手机号的一天内的全站总发送次数，超过限制，增加验证码

4 、 重置密码/ 密保

CSRF

测试方法：

    先登录一个帐号，然后直接访问一个构造好的带操作请求的连接，可导致执行了重置密保的操作。

防护方法：

    重置密保操作使用二次认证，或是限制 Referer，或是采取 token 验证。

验证步骤绕过

测试方法：

    进行验证时，随便输入，并且拦截服务器返回的包进行修改，欺骗浏览器（前端）使其以为通过验证，从而绕过验证的步骤。

防护方法：

    判断是否通过验证，应该在后端进行判断，而不是在前端进行判断。

重置他人密码/ 密保

测试方法：

    使用自己已通过验证得到的 token，去设置他人的密保。
防护方法：

    将 token 和帐号进行绑定，一个 token 只能对应一个帐号。

5 、 充值支付

篡改支付金额

测试方法：

    购买一定金额的游戏币，交易时拦截包把金额改为很小进行支付。

防护方法：

    完成交易前后端判断支付的金额是否和游戏币对等。

篡改商品数量

测试方法：
    购买一定金额的商品，交易时拦截包修改商品数量为较大的数，进行交易但是价格不变。

防护方法：

    完成交易前后端判断支付的金额是否和商品数量总值对等。

无限制创建订单号

测试方法：

    创建订单时，抓包并写脚本对该包进行多次重放，从而创建大量订单，造成资源占用和浪费。

防护方法：

    对一个用户一定时间内能创建的订单数进行限制。

支付金额为负数导致余额增加

测试方法：

    支付时将金额或商品数量改为负数，造成边界溢出，导致购买商品不用花钱甚至增加自己的余额。

防护方法：

    交易时后端检查判断数值的范围。
